Layout of the Documentation

The documentation might seem complex at first, but it is really very simple. This section will give you an idea of how the documentation is laid out, so you can easily work with it.

manual.xml is the big cheese for all the documentation. It defines the chapters and is used by the XML parser to generate the docs. If you need to add a new chapter, this is the file to see.

The biggest chunk of the documentation is stored in the sections/ directory. Unless it is a general file on the top level of the manual, files should be contained in sub-directories in the sections/ directory. For instance, Gallery 1.x files are located in the gallery1/ directory in the sections/ directory.

In each of the sub-directories of sections/, there will be an index.xml file, which defines the layout for the chapter. Normally, this file just contains entity references to all the sections in the chapter (see below). For example, the index.xml for the guide-standards/ directory looks something like this: 


<?xml version="1.0" encoding="ISO-8859-1"?>
   
 &chapters.guide-standards.basics;
 &chapters.guide-standards.formatting; 
 &chapters.guide-standards.generating;
 &chapters.guide-standards.layout;

Where these entities come from is described below.

Most every file used in the documentation can be referenced by an entity (as shown above). This allows us to easily include the documentation. File entities are described in the files.ent file in the base directory of the docs module. The file is commented and should be easy to understand.

Other entities, such as notes that will appear in the documentation often, should appear in the global.ent file. Entities should be defined here if they appear in multiple places in the documentation, so they can be updated in one place but automagically affect multiple places in the documentation.
