Naming Conventions

Naming conventions are important in maintaining a documentation system like this. This section discusses naming conventions for the id parameters of tags such as <chapter>, <sect1>, <sect2>, etc and entity names for files... It is important to maintain consistency in order to be able to link these documentation together efficently.

In the files.ent file, entities should be named something like this &chapter.chapter-name. brief-section-name. For example, this section would be named &chapters.guide-standards.naming-conventions; The brief-section-name part is left off when referencing the index.xml file in each section. For example, the entity referring to sections/guide-standards/index.xml would be named &chapters.guide-standards;.

Each major section should have an id attribute that it can be referenced by. The id attribute allows authors to link the documentation together by referencing the value of this attribute. The id attribute should be exactly the same as the entity in files.ent, except without the &chapter. For example: 


<?xml version="1.0" encoding="ISO-8859-1"?>

 <sect1 id="guide-standards.test-file">
  <title>Test File</title>
  
  <para>
   I am a paragraph
  </para>
  
  <sect2 id="guide-standards.test-file.another-section">
   <para>
    I am a section 2 tag.  My id attribute is derived from my parent's id
    attribute.
   </para>
   <sect3 id="guide-standards.test-file.another-section.yet-another">
    <para>
     Just another example, ma'am
    </para>
   </sect3>
  </sect2>
 </sect1>

Note

<sect2>, <sect3>, etc.. tags don't need to have an entity in files.ent, unless they are included in their own file (which is usually not the case).
